# Client Specification

## Scope

This document covers client-owned architecture and runtime behavior. Shared contracts such as coordinates, LODs, and WebSocket event names live in `docs/architecture-overview.md`.

## Entry Points

- `src/client/main.tsx`: creates the shared React Query client and renders `App`
- `src/client/app/App.tsx`: thin root that renders `WorldViewPage`
- `src/client/app/components/WorldViewPageContent.tsx`: app composition root that wires controls state, HTTP data, WebSocket invalidation, scene callbacks, sharing, and HUD chrome
- `src/client/features/world-view/components/World3DView.tsx`: imperative Three.js runtime boundary
- `src/client/features/world-controls/WorldControlsProvider.tsx`: reducer-backed control state, persistence, and fly-to requests

## Ownership By Area

- `src/client/app/`: compose features together; keep cross-feature wiring here
- `src/client/features/world-controls/`: voxel scene layer visibility, graphics/debug settings, chunk stats/loading breakdown, loading overlay, and `localStorage` persistence
- `src/client/features/world-view/`: data hooks, WebSocket hook, scene/runtime code, terrain and voxel loading, labels, markers, and the browser worker
- `src/client/lib/`, `src/client/hooks/`, `src/client/types/`, `src/client/utils/`: shared client infrastructure used across features

## Runtime Model

- React Query is the source of truth for HTTP data. `main.tsx` sets `staleTime: Infinity`, so refresh is event-driven rather than focus-driven.
- The world viewer initializes as a voxel scene for every page load. Legacy `mode` URL parameters are ignored, and the HUD does not expose a terrain/voxel selector.
- Copied location URLs are camera-only: they include position, zoom, theta, phi, and `focus=exact` parameters, and they do not include world-view mode state. `focus=exact` restores the supplied target coordinates without surface correction.
- URLs with `x`, `y`, `z`, `zoom`, `theta`, and `phi` but no `focus=exact` marker remain valid as map-compatible links. The startup camera treats their `z` value as a hint, focuses the best loaded visible surface at the supplied `x,y` with voxel geometry preferred over terrain, preserves `zoom`, `theta`, and `phi`, and retries after nearby terrain or voxel meshes load if no surface is available immediately.
- `useWorldData()` always loads world metadata, the surface index, the save block palette, and the voxel chunk index during initial page load so voxel rendering prerequisites are available immediately.
- `useWebSocket()` maintains the `/ws` connection, and `useWorldViewRefreshSubscriptions()` maps socket events to query invalidation.
- `WorldControlsProvider` owns low-frequency UI state and persists graphics/layer settings through `src/client/lib/world-view-storage.ts`; older stored versions are discarded and the app falls back to defaults.
- Atmosphere and block-emissive lighting are client-local presentation state under the existing graphics/debug settings. The default balanced atmosphere uses a fixed midday time-of-day, a Cubyz-stylized CSS sky gradient behind the transparent WebGL canvas, atmosphere-colored ambient/hemisphere/sun/fill lights, restrained fog-like depth enhancement, and balanced block-light quality. Atmosphere quality `0` disables both atmosphere-driven local block lighting and the atmosphere layer, restoring the fixed dark background and previous midday-style light baseline. Night atmosphere keeps a low-intensity ambient/skylight floor (dark blue ambient/hemisphere/sky/fog endpoints rather than near-black) so terrain silhouettes and vegetation remain readable against local emitters without making night look like daytime.
- Chunk stats are published continuously from the scene runtime so loading UI can stay accurate even when debug overlays are off.
- The loading overlay appears immediately when work starts, stays visible for a short linger after work completes, and uses a compact green cube on mobile.
- `World3DView.tsx` keeps scene state in refs and delegates most runtime work to `features/world-view/lib/`; avoid moving per-frame state into React state.
- Voxel loading uses the staged, bounded scheduler described below plus warm caches to avoid rebuilding everything during camera movement. Fine-detail requests are gated by the advertised voxel chunk index, so an unavailable child region cannot be requested merely because the camera is close. Terrain loading remains available for the optional terrain underlay and terrain-derived labels.
- Voxel LOD selection uses 3D effective distance from the camera to each region, using loaded voxel local top-height data at the camera X/Y as the reference surface Z and falling back to spawn Z before nearby tiles are available.
- `src/client/features/world-view/workers/voxel-mesh.worker.ts` decodes complete base geometry off the main thread before frame-budgeted Three.js upload, then optionally handles separately scheduled emissive enhancement.
- The voxel worker reads the cache-versioned mixed record payload. Parametric greedy cube records use `face`, `plane`, `u`, `v`, `du`, and `dv` rectangle fields, while fractional model/semantic records use fixed-point `u32` X/Y/Z vertices in `1/4096` voxel-cell units. Both paths convert to world coordinates before normals, colors, bounds, and chunk-top-height coverage are calculated.
- The optimized voxel worker uses a counting pass and then writes directly into opaque and transparent quadrant base arrays without constructing the emitter grid. If enhancement is enabled and applicable, the base result transfers the original compact buffer back without cloning; a later deterministic traversal returns only non-empty opaque quadrant normalized emissive arrays. The legacy explicit-vertex decoder remains isolated for stale development payloads only.
- Main-thread and browser-worker messages use the typed cancellable protocol documented in `docs/architecture-overview.md`. Production optimized decoding runs through an asynchronous checkpoint driver while the synchronous `buildMeshArrays` entrypoint remains available for deterministic test decoding.
- The voxel worker decodes per-quad block palette indices and render-kind values, splits opaque and transparent quads into separate quadrant mesh arrays, and preserves palette indices as per-triangle metadata on both streams. Cursor raycasts use `intersection.faceIndex` to resolve the selected voxel triangle to a palette index and then to a block ID from the loaded save block palette.
- The voxel worker decodes only the server's represented-source emitter records into world-space block-light inputs; the binary decoder, VXM6 metadata defaults, and block-light quality controls are unchanged. Qualified halo records remain payload-local bake inputs, while only qualified own LOD `1` records can create runtime glow or point-light accents. Coarse own records are aggregate centroids rather than physical source blocks, so they remain mesh-local bake inputs but never create runtime accents.
- `npm run test:voxel:client` imports the production worker's mesh-building path under Node with a minimal worker-global shim and verifies deterministic decoded/baked output plus in-radius, out-of-radius, and restricted-open-face emitter influence without a browser or WebGL.
- `npm run test:core:client` verifies terrain 3 by 3 gutter invalidation and voxel live-update cancellation/refresh state without rendering a scene. `npm run test:core:terrain` passes temporary adjacent `.surface` fixtures through the production terrain route and mesh builder to compare shared edge positions and normals.
- `npm run test:voxel:contract` feeds adjacent generated LOD `1` payloads through that production worker. The capped fixture distinguishes same-edge-distance halo sources by Y/Z locality, while the asymmetric Y-boundary fixture proves matching own/halo source retention despite different unrelated own-emitter populations. Both key opaque seam vertices by world position plus normal, normalize the compact emissive attribute, and fail when any matched channel differs by more than one `Uint8` encoding step. It also covers adjacent LOD2 production-summary payload records, where neighbor-owned coarse records participate in the bake through halo semantics but are excluded from returned emitter ownership. `npm run bench:voxel:client` reports serial warmup/sample timing, output size, and worker bake metrics separately from ordinary correctness tests.
- Mesh-local emitted light is primary. Per-emitter gain starts from `sqrt(power)`, caps at `8`, is limited by `max(1, 3 - (radius - 20) / 4)`, then uses `min(1, (20 / radius)^e)` radius-energy attenuation. LOD 4 additionally caps compressed gain at `0.75` to avoid a concentrated mid-detail brightness spike. `e` is 6 at LOD 8, 1 at LOD 16 and 32, and 6 below LOD 8, so broad representatives trade peak intensity for footprint at each sampling density. At LOD 16 and 32, the worker expands the index/falloff radius by half the coarse-cell diagonal (capped by the 64-block wire safety limit) to cover payload-cell center quantization; gain still uses the encoded source radius. Existing open-face attenuation, smoothstep falloff, wrapped lambert, deterministic candidate cap, per-channel clamp, receiving-surface hue bias, transparent exclusion, and reduced model-quad multiplier remain. Default power `1`/radius `12` follows the previous LOD 1 calculation unchanged.
- The radius-aware numeric index inserts each representative into every cell intersecting the same quantization-padded radius used for falloff, plus one cell of boundary slack on every side. Every vertex scans its complete fixed 3 by 3 by 3 neighboring-cell footprint with allocation-free de-duplication, so own versus halo ownership and unrelated primary-cell records cannot change which eligible seam emitters are considered. Emitters exceeding the 512-cell fan-out budget enter a bounded broad-emitter fallback checked conservatively. Vertex queries reject out-of-radius and face-blocked candidates before selecting the nearest deterministic 32 reachable candidates (squared distance then payload index), so insertion slack cannot crowd out a reachable halo emitter. Quad culling expands by the same one-cell probe margin, checks broad bounds separately, and preserves false-positive behavior for excessive scans. Unlit quadrants still omit emissive attributes and compact normalized integer output remains unchanged.
- Each received terrain-update batch is expanded and unioned once through the shared voxel invalidation mechanics before refresh versions, cancellation, warm eviction, or direct refresh. LOD `1` updates cover their emitter halo plus aligned LOD `2` through `32` ancestors; coarse updates cover their same-LOD emitter-summary halo. Loaded stale geometry remains visible and selectable until a current-version mesh atomically replaces it. Voxel add, removal, warm-cache restore, and same-key replacement increment a loaded-voxel revision. On the next rendered frame, `BlockLightRuntimeManager` reconciles emitter metadata only when that revision changed; it does not scan loaded tiles, synchronize regions, or select accents on animation ticks skipped by the active/idle frame cap.
- The render loop enters the configured idle frame cap after loading and dirty scene work drain and no pointer, keyboard, or camera interaction occurs for the idle delay. A stationary pointer may remain over the map while idle; pointer movement or an active drag immediately restores the active cap.
- The shared opaque voxel material is patched (`features/world-view/lib/block-light-mesh.ts`) so worker-baked `emissiveLight` is added as emissive radiance scaled by one shared `blockLightStrength` uniform. The compact normalized integer `emissiveLight` array is uploaded as a normalized `BufferAttribute` so the shader still reads a `vec3` in `0..1` with no shader contract change. On every rendered frame, the uniform follows the configured daytime-floor-to-night ramp and becomes `0` whenever block-light quality or atmosphere quality is `0`; skipped ticks need no update because they produce no frame. The bake and uniform require no Three.js point lights, so any number of emitters combines deterministically.
- `BlockLightRuntimeManager` retains own-emitter metadata per loaded tile but selects runtime accents only from LOD `1` tiles. It owns one fixed global pool of `HIGH_GLOW_BUDGET` reusable sprites/materials (220), plus a point-light pool that grows only to the configured high-quality budget. Active rendered frames select at most the current glow/point-light budgets with bounded nearest-camera insertion using squared distance and stable source iteration for ties, then update slot position, color, scale, opacity, and visibility; decoded emitter or tile count never creates tile-owned sprite groups. Opacity uses capped fourth-root power, and optional point-light intensity/distance use capped square-root power/radius; power `1`/radius `12` preserves prior defaults.
- Glow sprites and point lights are secondary runtime accents, not the illumination source baked into geometry. Point lights may dynamically affect Lambert-lit voxel base color at high quality, but they neither write nor alter worker-baked emissive attributes or the shared emissive-strength uniform. When daytime, block-light quality `0`, atmosphere quality `0`, or a zero accent budget makes accents inactive, the transition hides pooled sprites and point lights once while retaining the configured daytime mesh-emissive floor when lighting remains enabled; steady inactive rendered frames update the uniform and stats without flattening emitter metadata or traversing the glow pool again.
- Temporary voxel-lighting performance diagnostics live in the debug parameter panel ("Diagnostics" section). "Progressive Voxel Meshing" defaults on and can select the retained one-phase base-plus-emissive fallback for correctness and cached-payload benchmark comparison. "Diag: Halo Emitters" adds `?halo=0` so payloads omit neighboring-region halo records, while "Diag: Emissive Attributes" skips enhancement entirely. Fetch demand governs base work, while progressive attachment preserves visible base geometry and remains valid after the base becomes fresh and leaves fetch demand when its loaded tile is fresh for the same refresh version and exact base identity. Refresh, unload, replacement, staleness, or warm-cache movement rejects the result. Mesh-local emissive attributes remain separate from runtime glow sprites and point-light accents, so their independent accent behavior is unchanged. Cached-payload benchmarks compare base-visible time, enhancement time, total progressive CPU, retained compact bytes, and base/enhancement output bytes against one-phase output.
- Contributor light comparisons use `npm run validate:voxel-lighting -- --lod <lod>` against a fixed camera URL. The harness seeds the fixed nighttime (`atmosphereTimeOfDay: 0`) balanced graphics settings before page load, with `voxelLod1MaxDist: 1200` so the shared `zoom=500` viewport can select LOD 1; it intercepts `/api/world/chunk-index` and retains only the requested LOD. After its settle period, it rejects the capture unless the `Loaded by LOD` HUD row has one or more target tiles and zero tiles at every other supported LOD. `minRenderedVoxelLod` is a detail floor, not a force-LOD control, and must not be used as capture isolation. The enabled image uses balanced block-light quality and worker emissive attributes; the disabled image sets both `blockLightQuality` and `voxelEmissiveAttributesEnabled` to `0`, and must report zero accent emitters and emissive bake output. The harness saves viewport PNGs, structured settings/HUD/diagnostic metadata, and grayscale absolute-delta mean plus the 2% threshold footprint under the ignored `artifacts/voxel-lighting-captures/` directory.
- The debug chunk stats separate loaded voxel geometry from retained CPU metadata, including the two additional emitter numbers, and queued output now counts emissive arrays and emitter records. Server diagnostics expose metadata bytes, power/radius ranges, and summary work; client emissive diagnostics retain grid-build, bake, and evaluated/culled quad interpretation for validating radius-aware cost.
- Server-generated Cubyz rotation semantic geometry uses the same voxel binary payload and fixed-point decode path as static non-cube model quads; no browser-side Cubyz asset loading or semantic decoding is required.
- Opaque voxel meshes use the existing baked vertex-color material path. Transparent voxel meshes use a separate material with transparent blending and `depthWrite` disabled so opaque geometry behind glass remains visible with an approximate accumulated tint.
- The voxel worker excludes transparent top faces from chunk-top-height coverage used by terrain-underlay and label behavior, so transparent-only glass roofs do not fully occlude the terrain underlay.
- The cursor HUD shows `X/Y/Z` coordinates for terrain and voxel hits. In advanced mode, rendered voxel mesh hits with a resolvable palette index show the saved block ID on a second row before LOD/region debug details; terrain underlay hits, missing palette mappings, and non-advanced mode remain coordinate-only.
- Voxel AO is split between server and client responsibilities: the server packs per-quad AO for LOD `1/2` top faces and concave vertical wall corners, while the client applies the final wall shading directly, uses separate debug intensities for top and wall AO, and still softens top-face AO near active LOD boundaries after visibility selection.
- Sign text is fetched per region through a dedicated React Query fetch keyed by `["signs", lod, regionX, regionY]` (`features/world-view/lib/sign-records.ts`), gated to LOD `1`. `SignLayerManager` (`features/world-view/lib/sign-layer.ts`) is an imperative scene-runtime layer that builds one texture-mapped quad per sign from the server-provided text-plane corners and reconciles rendered regions with the set of loaded LOD-`1` voxel regions.
- Sign text quads are built only at LOD `1`; when the active LOD moves away from `1` all quads are removed, and they are rebuilt from cached records when it returns. Each quad uses a nearest-filtered `THREE.CanvasTexture` painted from a 128x72 canvas (4px margin, `Unscii16` at native pixel size, transparent background, 16px line height, clipped past the 64px usable height) and a depth-tested material so terrain occludes the text. Quads are nudged slightly along the sign normal to avoid z-fighting.
- Before layout, the client parses the sign record's `text` as Cubyz formatted source text using `graphics.TextBuffer.Parser` semantics in `features/world-view/lib/sign-texture.ts`: `*` toggles italic, `**` toggles bold, `__` toggles underline (single `_` remains visible), `~~` toggles strikethrough (single `~` remains visible), `\` consumes itself and renders the next codepoint literally, `#` consumes up to six following hex codepoints as color nibbles (non-hex input uses `0`) to set the current color, and `§` resets bold/italic/underline/strikethrough while preserving the current color. Initial color is black. Formatting controls do not render as visible glyphs.
- Sign text wrapping measures visible characters only using canvas `measureText` with the active per-character font style (so bold/italic advances are measured correctly), splits on explicit `\n`, greedily word-wraps at the 120px usable width, hard-breaks single words wider than the usable width, centers each visible line, and clips past the 64px usable height.
- Sign text drawing maps color to `ctx.fillStyle`, bold/italic to canvas font-weight/font-style variants (browser-synthesized Unscii approximates the Cubyz shader effects), and underline/strikethrough to 1px `fillRect` runs in the active segment color across each styled segment's measured width (underline 15px below the line top, strikethrough 8px below). No drop shadow is drawn. The `CanvasTexture`, transparent background, and sign quad lifecycle are otherwise unchanged from the geometry-only phase.
- Sign resources (canvases, textures, geometries, materials) are disposed on region unload, sign-record change, LOD change away from `1`, and scene teardown; no per-frame sign state lives in React. Sign records are invalidated on `world-updated` (all regions) and `terrain-updates-batch` (affected regions).
- Player marker model assets are loaded lazily per avatar model ID when player markers are needed. The client may receive arbitrary `entityModelId` strings from `/api/players` (the server returns any palette-resolved ID verbatim and does not pre-filter to a supported set), so `World3DView.tsx` collects the distinct `entityModelId` values among current players (plus the default `cubyz:snale`) and `lib/avatar-assets.ts` requests `/api/assets/player-marker/:entityModelId`, loads the manifest-provided GLB with `GLTFLoader` and texture with `TextureLoader`, and caches per-avatar load state, normalized template, active texture, and inactive grayscale texture in an imperative map keyed by `entityModelId`. A per-manifest `available: false` response causes a per-avatar fallback rather than a global failure.
- `syncPlayerMarkers` renders each player with the marker model for their own `entityModelId`, falling back to the default `cubyz:snale` model when the player's own avatar is not loadable. Markers are recreated when a player's resolved avatar model ID, active/inactive state, or underground state changes.
- If an avatar manifest request fails, returns `available: false`, or the referenced GLB/PNG cannot be loaded (a per-model failure retries once), that player renders the default avatar or the existing fallback dot marker; unrelated players keep their own markers and the world view keeps working.

## View-Aware Voxel Refinement

Root regions remain eligible from unscaled 3D render distance; viewport/FOV screen-space scaling and view classification never hard-cull root coverage. Traversal refines an eligible root only as far as both the ordinary distance/screen-space thresholds and its view class allow. Root unload and in-flight cleanup use the same unscaled render-distance boundary, so zooming out to the maximum camera distance retains coarse map coverage.

- `focus`: if the resolved focus point is within half of the candidate region's world size from its horizontal bounds, the tile is locally focus-classified; it keeps the focus-distance refinement bias and is not coarsened by view relevance
- `forward`: conservative bounds intersect the expanded camera detail cone, so the distance, projected-size, LOD hysteresis, and minimum-rendered-LOD rules choose detail without an additional view clamp
- `peripheral`: the otherwise desired result is clamped at least one supported level coarser
- `rear`: the otherwise desired result is clamped at least two supported levels coarser

The classifier uses the camera's full 3D direction, vertical FOV, viewport aspect, and a conservative bounding sphere around the candidate bounds. Loaded and warm-cached tiles supply their actual horizontal region bounds and loaded min/max Z, including sampled top height where applicable; retaining known warm-cache bounds prevents unload/restore oscillation at vertical-distance or view boundaries. Candidates with no known geometry use their region bounds around reference surface Z with a vertical allowance equal to half the region world size. The forward cone uses the diagonal of horizontal and vertical half-FOV plus the candidate's angular radius, intentionally overestimating edge visibility.

Classification is stateful per tile. A tile enters forward detail with `voxelViewEnterMarginDegrees`; a previously forward tile remains there through the larger `voxelViewExitMarginDegrees`. Rear transitions likewise use distinct enter/exit tests around the camera hemisphere. This angular hysteresis combines with `voxelLodHysteresisRatio`, detail-request debounce, unload grace, and warm-cache restoration to avoid refinement churn while rotating.

Camera and controls-target positions are sampled during LOD polling. Movement beyond the fixed threshold refreshes the detail debounce timestamp; required coverage continues while optional detail remains deferred until `voxelDetailRequestDebounceMs` elapses. Pointer hover and tap positions do not directly change voxel LOD focus or trigger local refinement.

LOD transitions are gated by loaded scene coverage in both directions. During refinement, parent fallback remains visible quadrant by quadrant until available children actually provide coverage; a missing child in the authoritative chunk index is not requested, and the eligible loaded parent continues to cover it. During zoom-out or view-driven coarsening, an absent desired ancestor is requested while the selector discovers the coarsest non-overlapping frontier of already-loaded descendants beneath it. Those descendants remain visible until the ancestor enters `loadedVoxels`; queued fetch, worker, expanded-output, or warm-cache state is not scene-ready. Discovery never requests missing fine descendants or preserves obsolete fine work solely to complete fallback coverage. Once the coarse ancestor is loaded, it replaces the descendant frontier in one selection pass, and the normal grace period then unloads or warm-caches the fine resources. Coverage requests bypass the camera-motion detail debounce, while optional detail is committed only after the camera has remained stable for `voxelDetailRequestDebounceMs`.

## Voxel Work Priority

HTTP admission, compact-input dispatch, and base scene-ready insertion use the latest deterministic urgency comparator. Its order is:

1. Safety/urgency band: conservatively visible no-fallback holes, deadline-promoted focus base work, other coverage, then optional work.
2. View class: `focus`, `forward`, `peripheral`, then `rear`.
3. Phase: `base` before optional `enhancement`.
4. Capped continuous-demand age: older first within the same safe band.
5. Projected refinement benefit, effective 3D distance, LOD, then stable sequence.

Focus base work is deadline-promoted after 2 seconds of continuous demand (the 2.5-second deadline's 500-millisecond slack window), but never ahead of a visible-hole band. Every LOD selection pass increments the generation and reprioritizes queued fetching, compact-input, and scene-ready records without duplicating a tile phase or resetting `demandSince`; demand removal or refresh-version change resets that age. A visible stale-tile refresh derives safety and view priority from that tile's current loaded state and conservative bounds rather than using a global refresh priority. Already active fetch or worker work is not preempted solely for reprioritization.

A voxel `503 Service Unavailable` response is temporary capacity pressure only when it includes valid `Retry-After` guidance. The active HTTP slot is released immediately, the tile receives a per-key not-before deadline, and it does not consume the permanent generation-failure budget. At the deadline, only a tile that remains active demand (or a loaded stale refresh) re-enters the normal priority queue; other eligible work continues while it waits. Other HTTP and network failures retain existing retry accounting.

## Voxel Pipeline Tuning

The staged scheduler separates network, compact worker input, active worker execution, expanded worker output, and scene insertion. The following new `MapDebugSettings` fields and defaults are defined in `DEFAULT_MAP_DEBUG_SETTINGS`:

| Setting | Default | Implemented effect |
| --- | ---: | --- |
| `voxelCompactInputMaxJobs` | `8` jobs | Stops new fetch admission when that many completed compact inputs are waiting. |
| `voxelCompactInputMaxBytes` | `32 MiB` | Stops new fetch admission when queued compact buffers reach the byte boundary. Active responses may finish beyond it because response size is unknown at admission. |
| `voxelExpandedOutputMaxJobs` | `4` jobs | Stops worker dispatch while that many expanded results await scene insertion. |
| `voxelExpandedOutputMaxBytes` | `96 MiB` | Stops worker dispatch when queued output plus active prospective reservations reaches the byte boundary. |
| `voxelWorkerTarget` | `0` | Selects adaptive worker targeting; `1` through `4` selects a fixed target capped by the detected safety profile. |
| `voxelCancellationCheckpointMs` | `4 ms` | Worker time budget between cooperative yields that can observe cancellation. |
| `voxelViewEnterMarginDegrees` | `12 degrees` | Expands the conservative cone used to enter forward detail. Exposed as **View Detail Enter Margin** in the LOD debug controls. |
| `voxelViewExitMarginDegrees` | `18 degrees` | Retains the previous forward/rear class across a wider angular boundary; implementation clamps it to at least the enter margin. Exposed as **View Detail Exit Margin**. |

Compact and expanded byte limits allow one oversized item to proceed when its stage and reservations are otherwise empty, preventing a permanent stall. Already active HTTP requests can temporarily put multiple compact inputs over the byte limit; no new request starts until the stage drains. Before assigning each idle worker, the scheduler reserves an estimate from compact quad metadata and bounded actual-output history keyed by phase and LOD. Completion reconciles the estimate with actual queued bytes; cancellation, failure, or worker loss releases it once.

The adaptive pool does not turn `navigator.hardwareConcurrency` into a target. Fine-pointer desktop-class input starts at two workers and permits four; coarse-pointer/mobile-class input starts at one and permits two; an available device-memory hint below 4 GiB caps the pool at one; unavailable hints use the conservative one-to-two fallback. Bounded queue age, frame p95, worker busy/duration, scene backlog, reservation/output bytes, optional heap pressure, and recent camera-motion debounce state control the target. Healthy sustained demand adds at most one after cooldown. Pressure lowers the target promptly, while excess busy workers retire only after their valid work completes. Fixed one-worker mode uses the same priority, cancellation, reservation, and diagnostics paths.

Existing settings complete the backpressure chain: `maxConcurrentVoxelFetches` defaults to `8`; expanded output is consumed under `maxVoxelMeshesPerFrame = 8` and `voxelMeshBuildBudgetMs = 5 ms` per rendered frame. Pool target, compact/output capacity, cancellation checkpoint, and view margins are available in debug controls and persisted with the remaining map settings.

## Loading Diagnostics

The desktop advanced debug HUD has a **Voxel pipeline** section. A load generation starts at scene/world initialization and can be reset independently from LOD selection generations. Its bounded timing distributions each display p50, p95, maximum, and an independent valid sample count:

- **Base fetch duration**: fetch start to completed response body
- **Base compact-queue wait**: response completion to base worker dispatch
- **Base worker duration**: base worker start to completion
- **Base result-transfer wait**: worker completion to main-thread receipt
- **Base scene-ready wait**: base result receipt to completed scene insertion
- **Selection to base-visible state**: continuous demand selection to the first LOD pass that observes the inserted base visible; this is end-to-end and not additive with the preceding stage timings
- **Enhancement compact-queue, worker, result-transfer, and attachment waits** plus **selection to enhanced state**: optional populations measured separately from base-visible latency

Current-state gauges report compact-input and base scene-ready jobs/bytes; total queued demand; oldest continuous-demand age overall and grouped by LOD, safety, coverage, view, and phase; focus-base deadline misses; frame-work time; worker busy ratio and duration; reserved expanded bytes; and active/target workers. Observations also use bounded p50/p95/max distributions. The HUD separately retains cumulative cancellation and discard maps keyed as `stage:reason`; current reasons include demand removal, refresh supersession, shutdown, cancellation/result races, result validation, enhancement target mismatch, and scene validation. Worker and scene errors release scheduler capacity but are reported through the failed-region path.

The existing **Voxel Benchmark** section is worker-sample instrumentation, not complete end-to-visible latency. Required fetch/decode/total averages use total benchmark samples. Every nullable transfer, body-size, output, emissive, emitter, and server metric is aggregated with its own internal sum/count so sparse headers or optional phases do not dilute averages. The HUD exposes valid sample counts next to emissive grid-build, emissive bake, current-generation server run, and current-generation server halo averages; absent values display `n/a`. Cache hit/miss/unknown counts remain separate. Server halo averages exclude cached-generation halo timing.

## Change-Sensitive Facts

- Terrain invalidation is wider than a single tile: terrain payloads depend on the same-LOD 3x3 neighborhood because the server includes a 1-vertex gutter for seam-safe normals.
- If WebSocket event names or `terrain-updates-batch` payload shape change, update `useWebSocket()`, `useWorldViewRefreshSubscriptions()`, the server broadcaster, and docs together.
- The mobile/compact HUD reuses the same controls/debug/info content as desktop; loading overlay visuals may differ by viewport, but the visibility and progress source stay shared.
- Player marker model URLs are server-provided manifest URLs. Do not hardcode Cubyz asset paths in the client.
- Phase 1 atmosphere intentionally does not include water reflection/refraction, temporal anti-aliasing, cascaded shadows, PCSS soft shadows, raymarched fly-through volumetric clouds, or sun-shaft rendering. Those are future rendering phases that may reuse the viewer-local atmosphere state.
- If worker files or worker import paths move, run `npm run build`; the worker URL/bundling path is easy to break silently.
