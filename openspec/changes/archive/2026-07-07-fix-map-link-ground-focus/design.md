## Context

The 3D viewer accepts URL camera state with `x`, `y`, `z`, `zoom`, `theta`, and `phi`. Initialization currently applies those values directly to `OrbitControls.target` and computes the camera position from the supplied spherical offset. This preserves exact camera state, but it also means external map links that use a generic or stale `z` value can focus empty air rather than visible terrain.

In the observed repro, the URL supplies `z=64` while the terrain surface at `x=7649,y=4190` is around `z=30`. The camera is only three units from its target, so the viewer appears unable to zoom to ground level even though `OrbitControls.minDistance` is not the limiting factor.

The existing runtime already has surface-finding helpers that raycast loaded terrain/voxel geometry for spawn focus and world taps. The challenge is startup timing: URL camera state is applied before all nearby terrain or voxel meshes may be present.

## Goals / Non-Goals

**Goals:**
- Make external map/deep links focus a usable visible surface when the supplied target altitude is not near the terrain or voxel surface.
- Preserve exact restoration for links generated by the 3D viewer going forward.
- Keep the solution inside the client world-view runtime unless existing client data proves insufficient.
- Avoid pushing per-frame camera state into React state.

**Non-Goals:**
- Add new server endpoints or change terrain/voxel payload contracts.
- Redesign camera controls, input bindings, or OrbitControls behavior.
- Guarantee perfect surface resolution before any terrain/voxel data has loaded.
- Change coordinate conventions.

## Decisions

1. Add an explicit URL focus mode for viewer-generated links.

Viewer-created share URLs will include a mode such as `focus=exact` to indicate that `x,y,z` are an exact `OrbitControls.target` to restore. Links without this marker will be treated as map-compatible links whose `z` is a hint, not an exact focus target.

Alternative considered: Always preserve exact URL targets. This keeps old sharing behavior but does not fix the map-click failure. Alternative considered: Always snap every URL to terrain. This fixes map links but breaks exact camera sharing, especially for intentional above-ground or inspected voxel targets.

2. Resolve map-compatible startup focus against visible surface data.

For non-exact links, initialization should focus the camera on the best available visible surface at the requested `x,y`. The preferred source is loaded voxel/terrain geometry using the existing downward raycast approach. If no geometry is available at initial camera application, the viewer should perform a deferred correction after relevant terrain/voxel meshes become available.

Alternative considered: Fetch or parse surface height separately during URL parsing. That would couple URL parsing to world data and network behavior. Keeping resolution near the scene runtime preserves the existing boundary where camera state meets rendered world data.

3. Preserve the requested viewing orientation and distance when retargeting.

When a map-compatible link is retargeted from the supplied `z` to surface `z`, the camera should keep the requested `zoom`, `theta`, and `phi` so the user arrives with the intended view direction and scale, but around the corrected focus point.

Alternative considered: Use the default spawn camera offset after surface resolution. That would be stable but ignores the map link's requested camera state.

4. Treat legacy links without focus mode as map-compatible.

Existing links have no source marker. To solve the reported problem, absence of the marker should prefer surface-aware behavior. New viewer share links will become exact going forward.

Alternative considered: Use heuristics such as snapping only when `zoom` is small or `z` differs from surface by a threshold. This can reduce legacy share changes but makes behavior harder to reason about and still leaves edge cases where map links fail.

## Risks / Trade-offs

- Legacy viewer share links without a focus mode may restore to surface focus rather than the exact historical target. Mitigation: generate explicit exact links after this change and keep retargeting limited to non-exact links.
- Deferred surface correction could visibly move the camera after first render. Mitigation: run correction as soon as local terrain/voxel geometry is available and preserve orientation/distance so the movement is vertical retargeting rather than a full camera jump.
- Surface raycasts depend on currently loaded geometry. Mitigation: keep fallback behavior deterministic and retry correction when nearby terrain/voxel meshes are built.
- Retargeting to terrain may be wrong when a voxel structure above terrain is the intended focus. Mitigation: prefer voxel geometry when present, then terrain underlay, matching the visible-world tap behavior.

## Migration Plan

- No data migration is required.
- New share URLs include an explicit exact-focus marker.
- Existing URLs without the marker remain accepted and become surface-aware.
- Rollback is client-only: remove the focus-mode parsing/generation and deferred retargeting behavior.

## Open Questions

- Should the exact-focus marker be named `focus=exact`, `mode=exact`, or something more camera-specific such as `target=exact`?
- Should the client expose any debug indicator when a startup target was retargeted from supplied `z` to surface `z`?
