## Purpose

Define voxel lighting performance diagnostics so benchmark results separate halo timing from external-region load behavior and remain cache-safe across halo-enabled and halo-disabled request modes.

## Requirements

### Requirement: Diagnostics distinguish halo timing from external-region load behavior
Voxel lighting diagnostics SHALL report halo timing separately from aggregate external-region load/cache behavior so benchmark results can identify whether halo collection is still dominated by source data loading.

#### Scenario: Halo-enabled benchmark sample is recorded
- **WHEN** a voxel benchmark sample includes server metrics for a halo-enabled LOD 1 generation
- **THEN** diagnostics expose halo elapsed time and aggregate external-region cache/load counters for that generated payload or metrics interval

#### Scenario: Cached payload is served
- **WHEN** a voxel payload is served from cache instead of generated by a worker
- **THEN** diagnostics distinguish cached timing metadata from current-generation halo timing so averages do not misrepresent live halo work

### Requirement: Diagnostics remain cache-safe across halo modes
Diagnostic halo-enabled and halo-disabled requests SHALL remain distinguishable in cache behavior so benchmark samples do not mix payloads with different halo contents.

#### Scenario: Halo-disabled diagnostic request is generated
- **WHEN** a diagnostic request asks the server to omit halo emitters
- **THEN** the generated payload does not contaminate normal halo-enabled persistent voxel cache entries

#### Scenario: Normal halo-enabled request follows a diagnostic run
- **WHEN** a normal request is made after a halo-disabled diagnostic request for the same region
- **THEN** the server does not reuse the halo-disabled payload as the normal halo-enabled payload
